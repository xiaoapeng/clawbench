## Description
Piper's `PostResolve` callback replaces `cmd.Env` with `append(os.Environ(), ...)`, completely discarding any env vars previously set via `opts.Env` in `CLISpeechProvider.Synthesize`. The two env-setting paths are architecturally conflicting. Currently Piper doesn't set `opts.Env`, so there's no active data loss, but this is a correctness trap for any future provider that uses both `Env` and `PostResolve`.

## Impact
Latent bug — any future TTS provider that uses both `opts.Env` and `PostResolve` will silently lose environment variables. Could lead to broken functionality or security misconfiguration.

## Suggestion
In `PostResolve`, check if `cmd.Env` is already set and append to it rather than replacing with `os.Environ()`:
```go
if cmd.Env == nil {
    cmd.Env = os.Environ()
}
cmd.Env = append(cmd.Env, "LD_LIBRARY_PATH="+piperDir)
```
