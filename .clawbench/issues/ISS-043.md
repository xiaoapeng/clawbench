## Description
The `update` action in `ServeTaskByID` (scheduler.go:274) unconditionally sets `task.MaxRuns = req.MaxRuns`. The request struct declares `MaxRuns int` with Go's zero value default of 0. Unlike string fields which are conditionally updated (`if req.Name != ""`), MaxRuns is always overwritten. When a client sends a partial update (e.g., only changing the cron expression) without including `max_runs` in the JSON body, Go's JSON decoder leaves `req.MaxRuns` at 0, silently resetting the task's max_runs.

## Impact
- Any HTTP API partial update that doesn't resend the current `max_runs` value resets it to 0
- Tasks with `repeat_mode=limited` effectively become unlimited
- The CLI partially mitigates this by defaulting `--max-runs` to -1 and only including it when >= 0 (task.go:520-522), but the handler has no such guard
- Direct HTTP API calls (not through the CLI) are fully affected

## Suggestion
Use a sentinel value to distinguish "not provided" from "set to 0":
```go
// In the request struct:
MaxRuns int `json:"max_runs"` // Keep as-is

// After parsing, only update if explicitly provided:
// Option 1: Use a *int pointer
MaxRuns *int `json:"max_runs,omitempty"`

// Option 2: Use a sentinel value (-1 means "don't update")
if req.MaxRuns >= 0 {
    task.MaxRuns = req.MaxRuns
}
```
