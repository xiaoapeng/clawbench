## Description
Three concurrency issues in the TTS subsystem:
1. TTS goroutine captures *http.Request used after handler returns — accessing r after handler return is unsafe
2. SendTTSEvent can panic on send to closed channel — between Load and send, UnregisterTTSJob could close the channel
3. Concurrent TTS requests for same cache key cause job overwrite — second overwrites first, wrong events served to first client

## Impact
Panic/crash risk (send on closed channel). Data corruption from concurrent TTS jobs writing to same file. Wrong audio served to clients.

## Suggestion
1. Resolve localized strings before launching goroutine
2. Use recover() or sync.Once for channel close protection
3. Use LoadOrStore for atomic check-and-register
